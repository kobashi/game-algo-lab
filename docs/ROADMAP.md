# ロードマップと今後の方針

最終更新: 2026-07-17

Game Algo Lab を **トピック拡張可能な教材プラットフォーム** として育てる。  
Git でバージョン管理し、仕様・ドキュメントを先に固定してから分業実装する。

---

## 三本柱

| 柱 | ねらい | 主な成果物 |
|----|--------|------------|
| **1. トピックの拡大とカテゴリ化** | 経路探索だけでなくゲーム木・物理・設計などへ広げる | `docs/topics/CATALOG.md`、`js/main.js` の `TOPICS` |
| **2. 共通基盤（教材の統一感）** | どのトピックも同じ「学び方」で触れる | `docs/PLATFORM.md`、`css/`・`js/ds-viz.js` 等の共有部品 |
| **3. 分業体制** | 仕様が揃っていれば担当を分けて並行実装できる | `docs/WORKFLOW.md`、`docs/templates/` |

この3つは同時進行する。カテゴリを増やす前にプラットフォーム規約を書き、トピック着手前に SPEC を埋める。

---

## フェーズ

### Phase A — 経路探索の完成度（現行）

- **状態**: BFS / DFS / ダイクストラ / 最良優先 / A* が `ready: true`
- **直近の改善例**: 地図ペイント（コスト・壁・**複数ゴール G**）、DS 可視化、レスポンシブ UI
- **残作業の例**: 共通ペイント／盤面コードの抽出、地図エディタ体験の統一、バグ修正

### Phase B — 共通基盤の明文化と抽出

- `PLATFORM.md` に沿って **命名・レイアウト・操作・データ構造パネル** を固定
- デモ間で重複している盤面・ペイント・制御ループを **共有モジュール** へ寄せる（無理に全部一度にやらない）
- トップページを **カテゴリ単位** で見出し表示できるようにする（`badge` → カテゴリ ID）

### Phase C — ゲーム木シリーズ

カタログ順（実装もこの順を推奨）:

1. **AND-OR 探索** — ✅  
2. **Min-Max 探索** — ✅  
3. **α-β 法** — ✅  
4. **モンテカルロ法** — ✅ 2026-07-17 実装  
5. 多腕バンディット — 次  

各トピックは `docs/templates/SPEC.md` をコピーして SPEC を書き、レビュー後に実装ブランチを切る。

### Phase D — その他カテゴリ

- 物理・判定（AABB など）  
- 設計パターン（ステートマシンなど）  
- 受講生提案のミニゲーム連動（別枠で SPEC）

---

## カテゴリ（現時点の分類）

| カテゴリ ID | 表示名 | 例 |
|-------------|--------|-----|
| `pathfinding` | 経路探索 | BFS, DFS, Dijkstra, Best-First, A* |
| `game-tree` | ゲーム木 | AND-OR, Min-Max, α-β, MC, Bandit |
| `physics` | 物理・判定 | AABB |
| `patterns` | 設計パターン | ステートマシン |

詳細・状態は [topics/CATALOG.md](./topics/CATALOG.md) を正とする。  
メニュー表示の `badge` 文字列はカタログの表示名と揃える。

---

## 成功の定義（教材として）

1. **受講生**が同じ UI 習慣で新しいトピックに入れる  
2. **実装者**が SPEC + テンプレ HTML から迷わず着手できる  
3. **メンテナ**が PR で「プラットフォーム規約違反」を機械的に指摘できる  
4. GitHub Pages で常に **main = 公開可能な静的サイト** を保てる  

---

## やらないこと（当面）

- ビルドツール必須化（Webpack 等）。静的 HTML/CSS/JS を維持する  
- バックエンド・ログイン  
- 仕様なしの大規模リファクタ（共通化は小さく・トピック単位で）
