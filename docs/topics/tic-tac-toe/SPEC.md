## 1. 学習目標

受講生がこのデモのあとで言えること・できること:

1. 三目並べは**全探索で引き分けが証明できる**ことを、実際の探索で確認し説明できる
2. **α-β・メモ化・対称性除去**それぞれの ON/OFF で訪問局面数がどう減るかを**数値で**比較できる
3. **ランダムプレイアウト評価（モンテカルロ）と完全解の乖離**を、同一盤面で観察し説明できる

（発展）1〜9 から3つ選んで和 15 を作る「15ゲーム」との同型性を説明できる。

---

## 2. なぜゲームで使うか

固定の抽象木（`minimax` / `alpha-beta` デモ）から**実在のゲーム**への最初の橋。
ルール説明が不要な題材で「合法手の列挙 → 手の適用 → 終局判定 → 評価」という
ゲーム木探索の共通インタフェース（正本 §6.1）を初めて実ゲームに適用する。
4×4 オセロ（§6.3）で本格化する対称正規化・転置表を、8 対称・765 局面という
手頃なサイズで先に体験する。

---

## 3. アルゴリズム概要

- 入力: 任意の合法局面（既定は空盤）と手番
- 出力: 各合法手の理論値（勝ち/引き分け/負け）と最善手、訪問局面数などの計測値
- 停止条件: 終局（三目成立 or 満杯）
- 計算量の直感: 素の全探索は最大 9! 系のオーダー、到達局面は約 5,478、
  8 対称の除去後は **765**。削減が桁で見えることが教材の核。

```
Solve(board, player):
  if 終局: return 結果 (WIN/LOSE/DRAW を根の手番視点で)
  key = canonical(board)          # 対称性除去 ON のとき 8 変換の最小符号化
  if key in memo: return memo[key]  # メモ化 ON のとき
  best = -∞
  for move in 合法手:
    v = -Solve(apply(board, move), 相手)
    best = max(best, v)
    if 枝刈り ON and best が確定十分: break   # α-β
  memo[key] = best
  return best
```

MC 評価（比較用）: 各合法手から N 回ランダム終局し勝率平均。N を変えて収束と
「完全解とのズレ（悪手を過大評価する例）」を見せる。

## 4. 画面と操作

### 可視化で見せるもの

- 3×3 盤（クリックで着手・任意局面の編集）
- 各空きマスへの**評価値オーバレイ**（勝ち=緑 / 引き分け=黄 / 負け=赤）
- **計測パネル**: 訪問局面数・メモヒット数・カット数を、トグル構成ごとに横並び比較（表）
- **8 対称パネル**: 現局面の 8 変換と符号化値・代表値（canonical）のハイライト（正本 §6.3 の前振り）
- MC モード: 各手の勝率推定とプレイアウト回数、完全解との差の強調表示

### 必須操作

| 操作 | 挙動 |
|------|------|
| 解析 / 再生 | 現局面を解析し評価オーバレイと計測を更新（プログレッシブ表示） |
| 1ステップ | 探索を1展開ずつ進め、現在の探索枝と部分木を表示 |
| リセット / 再読込 | 空盤・計測クリア |

### パラメータ

| 名前 | 範囲 | 既定 | 説明 |
|------|------|------|------|
| α-β 枝刈り | ON/OFF | OFF | 素の全探索と比較 |
| メモ化（転置表） | ON/OFF | OFF | 同一局面の再訪を削減 |
| 対称性除去 | ON/OFF | OFF | canonical 化。765 局面を体感 |
| MC プレイアウト数 N | 10〜10000 | 100 | 収束の観察 |
| 速度 | スライダー | 中 | `createPlayback`、大=速（規約どおり `450 - v`） |

### データ構造パネル

- コールスタック（現在の探索経路の盤面ミニチュア列）
- 転置表（メモ）のサイズとヒット率
- トグル構成別の訪問局面数の比較表（`result-compare` 慣習に従う）

## 6. 成功 / 失敗の定義

- 成功時の表示: 空盤の解析結果が「引き分け」であること、および
  トグル 3 種の組合せで訪問局面数が単調に減る比較表
- 失敗にあたる状態はない（解析は常に完了する）。非法局面（O が先行等）は編集時に警告
- 比較表示: あり（トグル構成間・MC vs 完全解）

## 8. PLATFORM との差分

- グリッド地図（`map-format` / `bindMapPaint`）は使わない（盤面は専用 3×3 UI）
- ゲーム木の**全体 SVG は描画しない**（既存 game-tree 型と異なり木が大きい）。
  部分木（現在の探索経路＋子）のみの表示とする — `layoutTree` は部分木に適用
- それ以外（`mountTopicShellFromDataset`、`createPlayback`、`createStatus`、
  操作 id 慣習）は準拠
