## 1. 学習目標

受講生がこのデモのあとで言えること・できること:

1. 割り箸は**同一局面に戻る循環**を持つため、ゲーム「木」ではなくゲーム「グラフ」であり、
   通常の Min-Max（深さ優先の再帰）では解けないことを説明できる
2. 終局から遡る**後退解析（retrograde analysis）**で全局面が
   **勝ち / 負け / 引き分け（ループ）** の3値に分類される過程を、波（深さ）ごとに追える
3. **ルールのバリアント**（分割の可否・ちょうど5/5以上・mod 5）で必勝側や引き分け局面数が
   変わることを、同一 UI 上の比較で説明できる

（発展）将棋・チェスの千日手／同形反復ルールと引き分けラベルの関係を説明できる。

---

## 2. なぜゲームで使うか

三目並べ・ニム・オセロは**必ず終局へ単調に進む**（非循環）。実在のゲームには
状態が行き来するもの（位置取り、体力の回復、千日手）が多く、その最小の実例が
手遊びの割り箸である。局面数が高々 **15×15×2 = 450**（片手0〜4本・左右無区別）に
収まるため、**全局面の3値ラベリングを画面上に一望**でき、「探索できない構造を
逆向きに解く」という発想の転換を体験できる。

---

## 3. ルール（標準形とバリアント）

**標準形**（デモの既定）:

- 各プレイヤーは両手を持ち、指の本数は 0〜4。初期局面は互いに (1,1)
- 手番では自分の**生きている手**（1本以上）で相手の生きている手を叩く。
  相手のその手の本数 += 自分の叩いた手の本数。**5以上になったらその手は死ぬ（0本）**
- 両手が死んだ側の負け

**バリアント（UI トグル）**:

| トグル | 選択肢 | 既定 |
|--------|--------|------|
| 分割（split） | なし / あり（自分の両手の合計を任意に分け直す。ただし現状と同じ・左右入替だけの分配は不可） | なし |
| 死の条件 | 5以上で死ぬ / **ちょうど5**のみで死ぬ | 5以上 |
| mod 5 | OFF / ON（5以上は 5 で割った余りに戻る。0 なら死） | OFF |

分割ありは循環と復活を生み、引き分け（ループ）局面が現れやすい。
バリアントの組合せは実装が矛盾なく扱えるものだけ有効化する
（例: mod5 ON のとき死の条件は自動的に「余り0」）。

## 4. アルゴリズム概要（後退解析）

- 状態: `(自分の手 {a≤b}, 相手の手 {c≤d}, 手番)`。左右の手は区別しない
  （**ソート済み表現に正規化** — 三目並べの canonical 化の1次元版）
- 終局: 手番側の両手が (0,0) → 手番側の**負け**

```
RetrogradeAnalysis():
  すべての終局局面に LOSE（手番側敗北）を付ける      # 波 0
  繰り返し（波 k = 1, 2, ...）:
    未確定の局面 s について:
      ある手で LOSE（相手視点）に行ける  → s は WIN  （確定、波 k）
      すべての手が WIN（相手視点）に行く → s は LOSE （確定、波 k）
    新たに確定する局面がなくなったら終了
  残った未確定局面はすべて DRAW（ループ）
```

- 計算量: 状態 450 以下 × 遷移数（高々十数）で一瞬。**波が教材の主役**なので
  実装は波単位で途中状態を返せる形（ジェネレータ等）にする
- Min-Max との対比: 同じ局面に対し深さ制限付き Min-Max（深さ d）も実行でき、
  「深さを増やしても引き分け（ループ）局面の値が振動・未確定のままになる」
  ことを見せる（対比パネル。深さ 20 程度まで）

## 5. 画面と操作

### 可視化で見せるもの（2枚看板）

1. **対局ビュー**: 両者の手（指の本数を絵文字/図形で表示）。自分の手→相手の手を
   クリックで叩く。分割ありのときは分割 UI。各合法手に行き先局面の3値ラベルを表示
   （最善手ガイド）。CPU は最善応手（WIN 維持 / LOSE なら最長粘り、DRAW ならループ維持）
2. **全局面マトリクス**: 15×15 のセル（行=手番側の手ペア、列=相手側の手ペア）を
   勝ち=緑 / 負け=赤 / 引き分け=黄 / 到達不能=灰 で着色。**再生/1ステップで
   後退解析の波がマトリクス上に広がっていく**のを見せる（波番号を表示）。
   現在の対局局面をマトリクス上でハイライトし、対局ビューと連動させる

### 必須操作

| 操作 | 挙動 |
|------|------|
| 再生 / 1ステップ | 後退解析を波単位で進め、マトリクスの着色が広がる |
| リセット / 再読込 | 初期局面 (1,1)-(1,1)・解析クリア |
| バリアント切替 | 切替時に再解析。結果サマリ（W/L/D 局面数・初期局面の結論）を比較表に蓄積 |

### 計測パネル（result-compare 慣習）

- 現バリアントの 勝ち/負け/引き分け局面数、到達可能局面数、波の数
- バリアント間の比較表（切替のたびに1行追加）: 初期局面の結論が変わることを見せる

### データ構造パネル

- 現在の波で確定した局面のリスト（正規化表記 `(a,b|c,d)`）
- 深さ制限 Min-Max 対比: 深さ d スライダーと「未確定・振動」の表示

## 7. 成功 / 失敗の定義

- 成功時の表示: マトリクスが全局面 3値+到達不能で塗り終わり、初期局面の結論
  （バリアント別）が結果パネルに出る
- 3値ラベルは**不動点性質**を満たすこと（WIN ⇔ ∃手→相手LOSE、LOSE ⇔ ∀手→相手WIN、
  DRAW ⇔ 勝ち手なし ∧ ∃手→相手DRAW）— 検証スクリプトの必須項目
- 比較表示: あり（バリアント間・深さ制限 Min-Max との対比）

## 11. 未決事項 → 実装時の判断（2026-07-19 Sonnet5）

- **マトリクスの行列軸**: 「手番側/相手側」を採用（「先手/後手」の固定ラベルは不採用）。
  割り箸は左右対称のゲームで、先手/後手を分けて数えると状態が15×15×2=450になり冗長。
  手番側/相手側で正規化すれば 15×15=225 に収まり、後退解析のロジックも「手番側→相手側」の
  一方向遷移だけで閉じるため実装・説明ともに単純になる。
  行=手番側の手ペア、列=相手側の手ペア（`js/chopsticks.js` の `stateKey` = `mover|opp`）
- **分割ルールの細目**:
  - 合計0（両手とも死んでいる状態からの分割）はそもそも合法手0件＝終局なので発生しない
  - `(0,x)` など片手が死んでいる状態からの分割は**許可**する（例: 現状(0,3)→分割可能な相手は(1,2)のみ。
    (0,3)自身は「現状と同じ」として除外）。死んだ手を分割で復活させられることが割り箸のループ性の
    重要な源泉であり、これを禁止すると「分割ありでDRAWが出現する」というSPEC §7の成功条件を
    再現しにくくなるため許可を選んだ
  - 「現状と同じ・左右入替のみの分配は不可」は状態がソート済みペアで正規化されているため、
    実装上は「新しいソート済みペアが元と一致するものは除外」の1行で自然に処理できる
- **「ちょうど5」バリアントで合計が5を超える（6〜8）場合の扱い**（新たに判明した論点。§3には
  明記がなかったため実装時に決定）: 片手に物理的に5本を超える指を掲げることはできない、という
  制約により**不可能な手として除外**（`applyHitValue` が `null` を返し `legalMoves` から外れる）。
  もし「5を超えても死なずそのまま残る」と解釈すると、タップのたびに本数が際限なく増え続け得て
  状態空間が有限に収まらなくなり、SPEC §2 の「15×15×2=450以下」という前提が崩れるため不採用
- **CPU の「最長粘り」**: v1 では未実装（SPEC の許可通り）。WIN 局面では相手LOSEへの手、
  DRAW局面では相手DRAWへの手（DRAW局面には定義上必ず1つ以上存在することを検証済み）を選ぶが、
  LOSE局面（どの手を指しても理論上負け）では暫定的に先頭の合法手を選ぶ
- **終局判定の一般化**: SPEC本文の「手番側の両手が(0,0)」を、実装では「合法手0件」という
  より一般的な基準に拡張した。分割なしでは両者が同値だが、フル225状態のうち実際のプレイでは
  起こらない局面（例: 相手が既に両手死んでいるのに自分はまだ生きている、分割なしでタップ対象が
  いない局面）にも一貫した0件判定を適用でき、マトリクス全体を矛盾なく塗り分けられる
  （到達不能な局面は別途 `reachableStatesFrom` で灰色表示に分離するため、この一般化が対局UIの
  結果に影響することはない）
