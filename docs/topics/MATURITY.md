# トピック成熟度（修正状況）

最終更新: 2026-07-17  

各トピックが **一発で出したままか / 改訂を経たか / 授業用に据え置けるか** を揃えて記録する。  
あわせて **修正回数** と **更新日** をラベルに載せる。  
**正本の一覧は [CATALOG.md](./CATALOG.md)**。Web 表示の数値は **`js/platform/maturity.js` の `TOPIC_META`**。

---

## 成熟度ラベル（3段階）

| コード | 表示名 | 意味 |
|--------|--------|------|
| `oneshot` | **一発未調整版** | 要件どおり一通り実装した段階。大きな教材改訂・UI 再設計はまだない（one-shot） |
| `revised` | **改訂・調整版** | 公開後または実装後に、学習目標・操作・表示・バグ・基盤寄せなどで **意図した改訂** が入っている |
| `stable` | **安定版** | 授業・試作運用で据え置き可能と判断。**破壊的な仕様変更は避ける**（直すならパッチ級か次メジャーで明示） |

### 付帯情報（ラベルに併記）

| 項目 | フィールド | 意味 |
|------|------------|------|
| **修正回数** | `revisions` | 初回実装（一発）のあとに行った **意図した改訂の回数**。oneshot は通常 **0** |
| **更新日** | `updated` | そのトピック教材として最後に手を入れた日（`YYYY-MM-DD`） |

`ready: true`（メニューでデモを開ける）とは **別軸**。

| | oneshot | revised | stable |
|--|---------|---------|--------|
| デモが動く (`ready`) | 多い | 多い | 必須 |
| 修正回数 | 通常 0 | 1 以上 | 1 以上 |
| 破壊的変更 | 自由 | 可（理由を残す） | 原則しない |

## 何を「改訂」と数えるか（修正回数 +1）

**数える例**

- 学習上の弱点・比較 UI の追加（例: BFS のコスト比較、AABB の二重判定）
- 操作・ペイント・複数ゴールなど機能の追加・直し
- 説明レイアウトの作り直し
- バグ修正が教材の正しさに効くもの
- 初期地図・課題シナリオの意図した差し替え（学習効果のため）

**数えない例（更新日だけ変えてもよい）**

- ヘッダー/フッターの `topic-shell` 化だけ
- `createPlayback` への寄せだけ（挙動同じ）
- 文言の誤字1か所・CSS の微調整のみ（任意で +0 のまま更新日のみ）

**安定版（revised → stable）にするとき**

- [ ] 学習目標が SPEC / 画面で一通り辿れる  
- [ ] 主要な操作（再生・1ステップ・リセット）が破綻しない  
- [ ] 既知の致命バグがない（または注記済み）  
- [ ] 授業担当が「この版で進める」と決めた  
- [ ] CATALOG / `TOPIC_META` の成熟度を `stable` に更新し、日付を残す

## 更新ルール

意図した改訂のたびに:

1. **修正回数** を +1  
2. **更新日** をその日に  
3. 必要なら成熟度を `oneshot` → `revised`（または `stable`）  
4. 同じコミットで揃える:
   - [CATALOG.md](./CATALOG.md)（成熟度・修正・更新・改訂メモ）  
   - `js/platform/maturity.js` の `TOPIC_META`  
   - `js/main.js` の `TOPICS[].maturity`（コードのみ。回数・日付は `TOPIC_META`）  
5. 可能なら `docs/topics/<id>/SPEC.md` にも追記

## 初期分類の考え方（v0.9.0 試作時点）

| 成熟度 | 方針 |
|--------|------|
| 経路探索 5 本 | 改訂あり → **revised**（修正 2〜3） |
| ゲーム木 5 本 | 大規模改訂なし → **oneshot**（修正 0） |
| AABB | 二重判定など → **revised**（修正 2） |
| FSM | 一通り → **oneshot**（修正 0） |
| stable | 試作段階では **なし** |
