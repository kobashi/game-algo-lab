## 1. 学習目標

受講生がこのデモのあとで言えること・できること:

1. **盤面符号化**（局面→整数/文字列キー）と**転置表**で、合流する探索木の再訪を
   除去できることを、生の生成局面数との比較で説明できる
2. **8対称の正規化（canonical 値・手番込み）**でさらに局面が同一視されることを、
   「生の生成数 / 転置表後 / 対称除去後」の3段比較（正本 §6.3）で説明できる
3. **パスと終局**（両者連続パス）をゲーム木上で正しく扱う必要性を説明できる
4. 4×4 オセロの理論結果（先手/後手どちらが勝つか・最終石差）を、
   **自前の全解析**の結果として述べられる（文献値をあてにしない）

（発展）色交換対称。三目並べ（765局面）との規模差と、8×8 オセロが未解決である理由の想像。

---

## 2. なぜゲームで使うか

第2期ゲーム木シリーズの**最終段・集大成**。三目並べで覚えた転置表・8対称を、
パス・終局・石差という本物のルール複雑性と、**1桁以上大きい状態空間**に適用する。
探索空間削減テクニックが「あれば便利」から「**ないと厳しい**」に変わる境目を
計測で体感させる。

---

## 3. ルール（4×4 オセロ）

- 4×4 盤。初期配置は中央 2×2 に黒白2枚ずつ（標準オセロと同じ交差配置）。黒先手
- 手番は相手石を1個以上挟んで返せるマスにのみ着手可（8方向）。返せる石は全て返す
- 合法手がなければ**パス**。**両者連続パスで終局**。盤面が埋まっても終局
- 勝敗は石数の多い側。同数は引き分け。教材値は 勝ち/負け/引き分け の3値＋**石差**

---

## 4. アルゴリズム概要

```
Solve(pos):                     # pos = (盤面, 手番, 直前がパスか)
  if 終局: return sign(石差)     # 3値（石差も記録）
  key = encode(pos)             # 符号化: 手番を含める
  if 対称除去 ON: key = min(8変換の encode)   # canonical
  if 転置表 ON and key in TT: return TT[key]
  moves = 合法手; if 空: 手番交代してパス節点として再帰
  negamax（α-β ON なら窓、OFF なら全子）
  TT[key] = 結果
```

- トグル3種（α-β / 転置表 / 対称除去）は三目並べと同じ独立 ON/OFF
- **3段計測**（正本 §6.3 指定）: ①生の生成局面数（探索が展開した節点数）
  ②転置表による再訪除去後 ③対称正規化後 — の比較を主役の計測にする
- canonical は**手番を含む**こと・パス状態の扱いを正しくすること（正本の注意点）

### 性能ガードレール（重要・実装必須）

4×4 オセロは三目並べより状態空間が大きく、**全トグル OFF の素の全探索は
ブラウザを数秒〜それ以上ブロックしうる**。次を必須とする:

1. 実装前に Node で**初期局面からの素の探索の節点数を実測**し、報告に含める
2. 解析は**チャンク実行**（ジェネレータ＋`setTimeout`/`requestAnimationFrame` 等で
   進捗表示・UI 非ブロック）にする。三目並べの同期一括方式をそのまま流用しない
3. 素の全探索が重すぎる場合（目安 >3 秒）、8構成比較の既定は**中盤プリセット**
   （数手進めた局面）から実行し、初期局面での全 OFF 実行は「時間がかかる」旨の
   警告付きオプションとする。閾値・プリセットの選定は実測に基づき §11 に記録

## 11. 実装判断（実装後に確定・記録）

### 盤面表現・状態表現

- **盤面表現**: 16文字の string（`.`/`B`/`W`、行優先）を採用。ビットボード2枚（B用/W用の16bit整数）は
  高速化の余地があるが、性能実測（下記）で素の全探索が初期局面から約0.4秒と十分高速だったため、
  可読性・三目並べ／割り箸との一貫性を優先して string 表現に決定した
- **状態表現（pos = 盤面+手番+直前パス フラグの扱い）**: SPEC §4 擬似コードの pos = (盤面, 手番, 直前がパスか)
  のうち「直前がパスか」は実装の状態に持たなかった。理由: 終局条件「両者連続パス」は
  `legalMoves(board,'B').length===0 && legalMoves(board,'W').length===0`
  という**盤面だけから決まる性質**と数学的に同値である（パスは手番を渡すだけで盤面を変えないため、
  「両者ともこの盤面で合法手がない」かどうかは経路に依存しない）。これにより
  - 符号化 `encode(board,turn) = board + turn` は `(board,turn)` のみで正しく機能する
  - 転置表キー・canonical キーが余分な次元を持たずに済み、実装がシンプルになる
  - パスは「合法手が0のとき、手番交代した局面を評価して符号を反転する」という単一子の negamax 節点として
    自然に実装できる（`solveNode` 内の `moves.length === 0` 分岐）
  この設計判断の正しさは、ランダム対局500局・全手（102,320件）を独立実装と突き合わせて検証し、
  パス・両者パス終局・満杯終局の3系統がすべて正しく発生・処理されることを確認済み（HANDOFF参照）。

### 性能ガードレール（SPEC §4 必須要件への対応）

- **実測値**（Node、初期局面から。詳細表は HANDOFF.md「4×4オセロ メモ」）:
  素の全探索（全トグルOFF）= 224,820局面・約0.4秒。α-βのみ=18,030局面・約37ms。
  α-β+転置表=11,822局面・約38ms。全ON=432局面・約6ms。8構成合計で約1.3秒
- **チャンク実行**（SPEC §4-2、条件なしの必須要件）: 実測が軽くても実装した。
  `solveNode` のジェネレータ版 `solveNodeGen`（`yield*` で再帰委譲、400訪問ごとに1回 `yield`）を書き、
  `js/platform/chunked-run.js` の `runChunked`（1チャンク16ms、`setTimeout` で継続）で駆動する。
  「解析」「8構成を比較」ボタンはいずれもこの経路でチャンク実行され、進捗（チャンク数・訪問局面数）を表示する
- **8構成比較の既定プリセット**（SPEC §4-3、条件付き要件）: 実測（初期局面0.4秒）が閾値3秒を下回ったため、
  中盤プリセットへの強制切替は行わなかった。8構成比較ボタンは現在選択中のプリセット（既定=初期局面）で
  そのまま実行する。この判断根拠（実測値）は上記の通り

### 3段計測とトグルの関係

- 「解析」ボタンは①現在の3トグル（α-β/転置表/対称除去）設定での1回の解析（訪問局面・転置表ヒット・カット表示）
  →②3段計測（①生=転置表・対称OFF固定 ②転置表=転置表ONのみ ③対称除去=転置表+対称ON、
  **α-βは現在のトグル値を3列共通で適用**）を連続実行する2段階構成にした
- 対称性除去は転置表がOFFだと訪問局面数に一切効果がない（canonical化した結果の格納先＝転置表が
  存在しないため）ことを性能実測で確認済み（8構成比較で「対称:ONのみ・転置表:OFF」の訪問局面数が
  「全OFF」と完全一致: 224,820=224,820）。3段計測のUIはこの依存関係が伝わるよう
  「②転置表後」から「③対称除去後」への追加削減として提示する

### 8対称パネルの platform 共通化

- **しない**と判断した。三目並べ（3×3・文字ピース）と4×4オセロ（4×4・円形コマ表示）は
  盤サイズ・ピース表現の両方が異なり、共通化すると設定オブジェクトでの分岐が増えて
  かえって見通しが悪くなると判断。両トピックとも「ミニ盤面グリッド＋代表値ハイライト」という
  **構造の型**は共有しているが、実装（`transformIndex`／描画）は各トピックの `js/<id>.js` に閉じたままとした
