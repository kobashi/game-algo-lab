## 1. 学習目標

受講生がこのデモのあとで言えること・できること（3つ以内）:

1. MCTS の **4相**（選択 → 展開 → シミュレーション → 逆伝播）を、1反復ごとに画面上で追い、各相の役割を説明できる
2. **UCB1 / UCT** が「多腕バンディットの腕選択」と同じ構造で、探索木の各節点で探索と活用をバランスすることを、式と木の成長の両方から説明できる
3. 同一の三目並べ局面で **MCTS 推定・素の MC 推定・完全解（`tic-tac-toe` solver）** を並べ、反復数 N を増やすと MCTS が完全解の最善手・評価に近づく様子を観察・説明できる

（発展）探索定数 C を変えると木の「横広がり／深掘り」がどう変わるか。  
（発展）4×4 オセロなどより深いゲームへの接続（本トピック v1 では実装しない）。

## 2. なぜゲームで使うか / なぜ三目並べか

- バンディットは「腕を引く」、MC は「固定木を乱択で評価する」。MCTS はその先で、
  **プレイアウトの結果で探索木そのものを成長させる**。
- 三目並べはルール説明が不要で、**完全解が既にある**（`js/tic-tac-toe.js` の negamax）。
  「サンプルで決める手法が、正解に収束するか」を同一盤面で検証できる（比較可能性・
  実在ルール優先。Fable5 レビュー指摘3・提案2）。
- 木が小さく 4 相と UCT の可視化に集中できる。深いゲームは v1 後の発展。

学習ストーリー上の位置:

```
抽象木 MC → バンディット(UCB1) → 三目並べ全解析(+MC) → 【本トピック MCTS】
                                              ↑ 題材・完全解を共有
```

## 3. ルール（題材）

三目並べ。`tic-tac-toe` と同一:

- 3×3、先手 X / 後手 O、交互に空きマスへ1石
- 縦横斜めの3目で勝ち。満杯かつ勝者なしは引き分け
- 局面表現: 9 文字（`.` / `X` / `O`）行優先（既存と同一）

**ルール実装は新規に書かない。** `js/tic-tac-toe.js` から import する:

| 用途 | 関数（既存 export） |
|------|---------------------|
| 合法手・着手・終局 | `legalMoves`, `applyMove`, `isTerminal`, `terminalValue`, `currentPlayer` |
| 乱択プレイアウト | `randomPlayoutOutcome` |
| 素の MC 比較 | `mcEstimateMoves` |
| 完全解比較 | `analyzeMoves`（全トグル OFF で十分。α-β 等は任意） |
| プリセット | `js/maps/tic-tac-toe-config.js` の `TICTACTOE_PRESETS`（少なくとも `empty`, `double-threat`, `endgame`） |

MCTS 固有の設定・木ノード型は `js/maps/mcts-config.js`（新設）に置く。

## 4. アルゴリズム概要

### 4.1 入力 / 出力

- **入力**: 合法局面（既定はプリセット）、手番、反復上限 N、探索定数 C、乱数シード
- **出力**:
  - 根の各子の訪問回数 `n`・累積報酬（または勝率推定）
  - 推奨着手（訪問最多、または平均報酬最大 — v1 既定は **訪問最多**。UI で切替可なら尚可）
  - 成長した探索木（可視化用）
  - 比較用: 完全解の各手の理論値、素の MC の各手勝率（同一 N・同一シード方針は §5）

### 4.2 報酬の定義（根の手番視点）

プレイアウト終了時:

| 結果 | 報酬 r（根の手番視点） |
|------|------------------------|
| 根の手番側の勝ち | `+1` |
| 引き分け | `0` |
| 根の手番側の負け | `0`（または `-1`） |

**v1 既定**: 勝ち `+1` / それ以外 `0`（勝率推定と一致し、UCB の解釈が「経験勝率」になる）。  
UI トグルで `+1/0/-1`（ゼロサム）に切替可能にするかは実装時判断（§11）。  
逆伝播時は **手番交代に応じて符号を反転**する（親は子の報酬の否定を積む）— ゼロサムの標準形。  
`+1/0` 方式のときは「現ノード手番の勝率」として格納し、子から親へは  
「親手番視点 = 1 − 子視点の平均」ではなく、**各ノードに「そのノードの手番側の累積勝ち」を持ち、  
プレイアウト結果を手番に応じて加算**する実装が読みやすい。  
**実装推奨（v1）**: 各ノードに `wins`（そのノードで手番だった側から見た勝利数）と `visits` を持ち、  
バックプロップで経路上の各ノードについて「その手番側が勝ったか」を判定して更新。引き分けは wins に足さない。

### 4.3 4相（1反復 = 1サイクル）

標準 UCT（Upper Confidence bounds applied to Trees）:

```
MCTS(root_state, N, C, rng):
  root = new Node(root_state)
  for i in 1..N:
    # 1. Selection — 未完全展開の葉（または未展開子がある節）まで UCB で降りる
    node = root
    state = root_state
    path = [node]
    while node が完全展開済 and not terminal(state):
      child = argmax_{a in children} UCB1(child, node.visits, C)
      node = child; state = apply(state, child.move); path.push(node)

    # 2. Expansion — 未試行の合法手を1つ子として追加（終局でなければ）
    if not terminal(state) and node に未展開手がある:
      move = 未展開手から1つ（v1: 合法手順の先頭、または乱択 — §11）
      state = apply(state, move)
      node = node.addChild(move, state)
      path.push(node)

    # 3. Simulation — 終局まで乱択プレイアウト
    outcome = randomPlayoutFrom(state, rng)   # 既存 randomPlayoutOutcome 相当
    # 4. Backpropagation — path 上の visits / wins を更新
    backup(path, outcome)
  return 根の子の統計と推奨手
```

### 4.4 UCB1（選択）

多腕バンディットと同じ形。子ノード i について:

$$
\text{UCB}_i = \hat\mu_i + C \sqrt{\frac{\ln N_p}{n_i}}
$$

- \(\hat\mu_i\): 子 i の経験勝率（`wins_i / visits_i`）。未訪問は \(+\infty\)（必ず一度は選ぶ）
- \(N_p\): 親の訪問回数
- \(C\): 探索定数。既定 **\(\sqrt{2} \approx 1.414\)**（理論的定番）。スライダー 0.1〜3.0

バンディットデモとの対応を lesson パネルに1行書く:  
「各子 = 腕、1 プレイアウト = 1 回引く、木の深さ方向に同じ選択が繰り返される」。

### 4.5 計算量の直感

- 1 反復 ≈ O(深さ × 分岐) 程度の軽量操作。N=1000 でも三目並べでは即座。
- 完全解の訪問局面（数千）とは別物: MCTS は **予算 N で近似**する。
- 空盤でも「推奨初手が中央・隅に寄る」「引き分け推定が支配的になる」を観察可能。

### 4.6 素の MC との差（教材の核）

| | 素の MC（`mcEstimateMoves`） | MCTS |
|--|------------------------------|------|
| 木 | 持たない（毎回ルートから一様乱択） | 訪問統計で木を成長 |
| 手の振り分け | 初手を均等に N 回ずつ、など | UCB が有望な枝に予算を寄せる |
| 深い手の連鎖 | 見ない | 有望変化を深掘りできる |

ダブルリーチ罠では、低 N の素の MC が外しやすい一方、MCTS は同じ N でも  
有望手に予算が集まりやすい — **同一 N・同一シードで横並び比較**する。

## 5. 画面と操作

### 5.1 可視化で見せるもの

1. **3×3 盤** — 現在の根局面。クリックで着手編集（`tic-tac-toe` と同様）。合法手に  
   MCTS の訪問数・勝率（または UCB）をオーバレイ
2. **探索木ビュー** — MCTS が構築した木（節点: 着手ラベル or ミニ盤、visits、\(\hat\mu\)）。  
   直近1反復の **Selection 経路をハイライト**し、Expansion で増えた節、Simulation の  
   結果色を短くフラッシュ
3. **4相インジケータ** — 現在の相（選択/展開/シミュレーション/逆伝播）をステータスまたはバッジで表示。  
   1ステップ実行時は相ごとに止まれると理想（§11: 最低限「1反復まとめて」可）
4. **比較パネル**（`result-compare` 慣習）:
   - 各合法手: MCTS 勝率 / 訪問数、素の MC 勝率、完全解ラベル（勝/分/負）
   - 推奨手の一致マーク（MCTS 推奨 == 完全解最善）
5. **計測**: 反復数、木のノード数、最大深さ、完全解との推奨一致（ブール or 手数一致率）

### 5.2 必須操作

| 操作 | 挙動 |
|------|------|
| 再生 | N 反復まで連続実行（`createPlayback`、速度スライダー準拠 `delayFromSpeed: (v) => 450 - v`） |
| 1ステップ | **1 反復**（4相一式）を進め、経路ハイライトを更新 |
| リセット | 木を破棄、統計クリア、局面は維持（またはプリセット再読込を明示） |
| 解析（比較） | 現局面で完全解 `analyzeMoves` と素の MC `mcEstimateMoves` を走らせ比較表を更新 |
| 盤面クリック | 空きへ着手 / プリセット読込 |

### 5.3 パラメータ

| 名前 | 範囲 | 既定 | 説明 |
|------|------|------|------|
| 反復数 N | 10〜5000 | 200 | 再生の上限。1ステップは常に1反復 |
| 探索定数 C | 0.1〜3.0 | 1.414 | UCB の探索項 |
| 乱数シード | 整数 | 7 | `mulberry32`。MCTS プレイアウトと素の MC で方針を揃える |
| 素の MC の試行 | （N と共通 or 別） | N と同じ | 比較の公平性のため **既定は MCTS の累計プレイアウト数と揃える** |
| プリセット | select | double-threat 推奨導線 | `empty` / `double-threat` / `endgame` 等 |

### 5.4 データ構造パネル

- 現在の Selection 経路（ルート→葉）をリストまたは `renderCallStack` 流用
- 根の子一覧: move, visits, wins, μ̂, UCB（選択中の計算値）

## 7. 成功 / 失敗の定義

- **成功（教材として）**:
  - 1ステップで4相の説明が UI だけで辿れる
  - `double-threat` で N を十分大きくすると MCTS 推奨が完全解の必勝手と一致する  
    （検証用の目安: seed 固定のもと N=500〜2000 程度で一致しやすいこと。実装時に数値を確定し SPEC §11 に記録）
  - 同一局面で比較表に完全解・MC・MCTS が並ぶ
- **失敗にあたる状態**: 非法局面は編集時に警告（`validateBoard` 再利用）
- 探索の「失敗」はなく、低 N で推奨が外れること自体が学習材料

## 13. 検証方針（実装時）

DOM 非依存の純関数として export し、Node または scratch スクリプトで:

1. 終局局面では展開せず、報酬だけがバックアップされる
2. N 回後に `root.visits === N`（または実装定義のカウントと一致）
3. 未訪問子が UCB で優先される（visits=0 が先）
4. `double-threat` + 固定 seed で、十分大きな N の推奨手 === `analyzeMoves` の最善手
5. 独立な素朴 MCTS（別コード）と統計の定性一致は任意（時間があれば）

`smoke-platform.py` のトピック登録チェックに含める。
