## 1. 学習目標

受講生がこのデモのあとで言えること・できること:

1. 1山の石取りで、負け局面を**終局から逆向きに着色**して全列挙し、
   **周期性（N mod (k+1) == 0 が負け）**を自分で発見・説明できる
2. 複数山ニムで、全探索の勝敗と **nim-sum（各山の XOR）≠0 ⇔ 勝ち** が
   全局面で一致することを確認し、2進表現での XOR 計算を説明できる
3. 「探索で解く」（節点数）と「構造で解く」（判定1回）の**計算量の差**を
   数値で説明できる — 「探索できること」と「理解して直接解くこと」の違い（正本 §6.2）

（発展）ミゼール規則（最後を取ったら負け）で理論解がどう変わるか。

---

## 2. なぜゲームで使うか

三目並べは「全探索で解ける」ことを見せた。ニムはその次の一歩 —
**全探索の結果に規則性を発見すると、探索そのものが不要になる**ことを見せる。
逆向き着色は割り箸の後退解析の**非循環版**（波が単純に深さ順に進む）であり、
ここで慣れておくと割り箸（循環あり）との対比が立体的になる。

---

## 3. ルール（2モード）

**モード1: 1山の石取り（サブトラクションゲーム）**

- 石 N 個の山がひとつ。手番で 1〜k 個取る（既定 N=21, k=3）
- **最後の石を取った方が勝ち**（正規形）
- 理論解: N ≡ 0 (mod k+1) が手番側の負け局面

**モード2: 複数山ニム**

- 山が複数（既定 3 山 = 3, 5, 7）。手番で **1つの山**から任意個（1個以上）取る
- 最後の石を取った方が勝ち（正規形）
- 理論解: nim-sum（全山の XOR）が 0 なら手番側の負け

ミゼール規則は v1 では実装しない（§11 発展・解説パネルで1行触れる程度）。

---

## 4. アルゴリズム概要

**逆向き着色（非循環の後退解析）**:

```
1山: value[0] = LOSE（取る石がない=直前の相手が最後を取った）
     n = 1..N: value[n] = (∃ t∈1..k, n-t≥0: value[n-t]==LOSE) ? WIN : LOSE
複数山: 終局 (0,0,..,0) = LOSE
     各局面 s: ∃手→LOSE なら WIN、全手→WIN なら LOSE（DAG なので DRAW なし）
```

**理論判定**:

- 1山: `n % (k+1) == 0` ⇔ LOSE
- 複数山: `xor(piles) == 0` ⇔ LOSE

**計測**: 逆向き着色/メモ化探索が触った局面数 vs 理論判定（剰余 or XOR 1回）。
全局面で **探索結果と理論判定が一致**することをデモ内で機械確認できる
（「一致確認」ボタン → 全局面照合の結果と件数を表示）。

## 5. 画面と操作

### モード1（1山）の可視化

- **0..N の帯**: 各局面を左から並べ、逆向き着色を 1ステップ（n を1つ）ずつ進めて
  W=緑 / L=赤 に塗る。塗り終わると **L が k+1 間隔で並ぶ縞模様**が現れる — ここが主役
- 縞の下に `n mod (k+1)` の値を並べ、剰余 0 と L の一致を視覚で確認
- 対局ビュー: 石をクリックで取る。CPU は理論解（剰余を 0 にする手）
- k スライダー（1〜5）: 縞の間隔が k+1 に変わるのを即時に見る

### モード2（複数山）の可視化

- 山と石の表示。手番で山から取る（クリック/ドラッグ）
- **2進パネル**: 各山の2進表記を桁を揃えて縦に並べ、列ごとの XOR（=nim-sum）を表示。
  手を指すたびに更新。nim-sum が 0/非0 で背景色を変える
- CPU は nim-sum を 0 にする手（存在すれば）
- 「一致確認」ボタン: 全局面（各山 0..初期値）で 探索 W/L と nim-sum 判定を照合し
  「N 局面すべて一致 / 探索節点 X vs XOR 判定 N 回」を result-compare に表示

### 必須操作

| 操作 | 挙動 |
|------|------|
| 再生 / 1ステップ | モード1: 逆向き着色を n 順に進める。モード2: 照合の進行表示（任意） |
| リセット / 再読込 | 初期局面・着色クリア |
| モード切替 | 1山 / 複数山 |

### パラメータ

| 名前 | 範囲 | 既定 |
|------|------|------|
| N（1山の石数） | 5〜40 | 21 |
| k（1山の最大取り数） | 1〜5 | 3 |
| 山構成（複数山） | プリセット (3,5,7) / (1,2,3) / (2,4,6)（偶勝ち例） | (3,5,7) |

## 11. 未決事項（実装時の判断・実施記録）

- **ミゼール規則**: 実装せず。学習目標4（発展）は `<details class="lesson-details">` 内の
  1行解説のみで対応（デモ化は次回改訂の候補のまま据え置き）。
- **モード2の「一致確認」の対象範囲**: SPEC の想定どおり、初期山構成の直積
  （各山 0..初期値）を対象にした。(3,5,7)=4×6×8=192局面、(1,2,3)=2×3×4=24局面、
  (2,4,6)=3×5×7=105局面（実装の `allStatesProduct` / `verifyAll` で実測・全数一致確認済み）。
  山数・上限を UI から変えられるようにはしなかった（プリセット切替のみ。v1 では過剰と判断）。
- **石の見た目・取り方の操作**: 石は絵文字ではなく丸ボタン（`●`）。取り方は
  「石をクリックすると、その石の位置まで残して取る」という位置指定方式に統一
  （1山・複数山どちらも同じ操作感にした。個数入力やドラッグは実装しなかった）。
- **モード切替 UI**: `<select>` を採用（タブではなく）。理由: 2モードで可視化パネル
  （帯 vs 2進パネル）が大きく異なるため、`data-mode-only` 属性で該当ブロックの
  表示/非表示を切り替える単純な実装にした。
- **CPU の着手アルゴリズム（モード2）**: メモ化探索の結果ではなく、nim-sum の
  XOR構成的性質（`piles[i] XOR nimSum`）から直接手を作る理論ベース実装にした
  （探索より高速で、かつ「一致確認」が探索と理論の等価性を別途機械証明しているため
  正当性は担保される。検証スクリプトで nim-sum≠0 の全局面に対しこの着手が必ず
  nim-sum=0 の局面へ遷移することを確認済み）。
