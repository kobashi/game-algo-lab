## 1. 学習目標

1. **前向き探索（S 起点）と後ろ向き探索（G 起点）を同時に進め、出会った点で経路を接合**できることを説明できる
2. 一方向 BFS と比べ、探索半径が両側に分かれるため **展開マス数を減らせる**場合があることを、同一地図の計測で観察できる
3. 複数ゴールがあるときは後ろ向きを **全ゴールからの multi-source BFS** にすると自然に扱えることを説明できる

（発展）辺の向きがあるグラフでは後ろ向きは逆辺が必要。本デモは無向グリッドのため対称。

---

## 2. なぜゲームで使うか

広いマップや長距離ナビで「スタート側だけ広げる」とフロンティアが円状に肥大化する。  
双方向はゴール側からも広げることで、**出会い地点付近の狭い帯**で接続でき、ゲーム AI の経路探索の高速化パターンとして定番。

---

## 3. アルゴリズム概要

- 入力: グリッド・壁・スタート・ゴール（1 個以上）
- 出力: 最少歩数経路（4 近傍・無向）、出会点、両側の展開数
- 停止: 前向き探索済み集合と後ろ向き探索済み集合が交差 / 両キュー空

```
Forward:  queueF, parentF, distF   — S から
Backward: queueB, parentB, distB   — 全 G から（multi-source）

while queueF と queueB が両方非空:
  # 既定: フロンティアが小さい側を1段拡張（交互トグル可）
  expand 1 node from chosen side
  if 拡張した子が反対側で既に訪問済み:
    meet = その子
    path = reconstruct(S→meet via parentF) + reverse(meet→G via parentB)
    return path
```

比較用: 同一地図で一方向 BFS の展開数を内部計算し表に並べる（別デモを開かなくてよい）。

---

## 4. 画面と操作

### 可視化

- 前向き訪問 / 後ろ向き訪問 / 各フロンティアを色分け
- 出会点を強調
- 最終経路（S→G）
- マスラベル: 前向き歩数 `→d` / 後ろ向き歩数 `←d`（片側のみなら片方）

### 操作

| 操作 | 挙動 |
|------|------|
| 再生 | 拡張を連続実行 |
| 1ステップ | 片側1ノード拡張（方策に従う） |
| 再読込 | 地図リセット |
| 拡張方策 | 小さいフロンティア優先（既定） / 交互（F→B→F…） |

### データ構造パネル

- 前向き Queue / 後ろ向き Queue
- 両側の探索済み件数
- 親ポインタは出会後に経路上を表示（任意）

- 出会点が経路上にあり、経路を接合できる
- 結果パネルに展開数（双方向合計 vs 一方向 BFS）を表示
