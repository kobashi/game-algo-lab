**総論**: JS 実装（コールスタック・α/β 窓・リグレット等の可視化）はどれも堅実。
問題は**初期データが浅すぎ・易しすぎて、実装済みの可視化が本来見せるべき現象を
発生させられない**ことに集中している。特にゲーム木 4 本は全て深さ 2 の同型木で、
oneshot 生成の典型的な弱点が出ている。

---

## 指摘一覧（重要度順）

### 1.【重大】α-β: 深さ2の木では βカットが構造的に発生し得ない

- 対象: `js/maps/alpha-beta-tree.js`（根 MAX → MIN → 葉 の 2 段）
- 実装 `js/alpha-beta.js` は `cut: 'alpha'|'beta'` を区別して表示する機能を持つ
  （`alpha-beta.js:63,221`）のに、**βカット（MIN の下の MAX で v≥β）は深さ3以上で
  しか起きない**ため、データが機能の半分を永久に殺している。
- 学生は「αカットだけがα-β法」と誤解しうる。α/β の**窓**という概念も、β が一度も
  効かないため下限 α の話に退化している。
- 正本 §6.1 の学習点「**手の順序で枝刈り量が変化する**」も、カットが1箇所しかない
  現行木ではほぼ観察不能。

**提案（具体木・手計算検証済み）**: 深さ3・葉12の木を新標準とする。

```
根 MAX ─ L(MIN) ─ L1(MAX): 5, 3        → L1=5
        │        └ L2(MAX): 7, 9       → 7≥β(5) で βカット（9 を刈る） L=5
        ├ M(MIN) ─ M1(MAX): 4, 2       → M1=4 ≤ α(5) で αカット
        │        └ M2(MAX): 1, 3       → 内部節点ごと刈られる（深い刈り）
        └ R(MIN) ─ R1(MAX): 6, 8       → R1=8
                 └ R2(MAX): 7, 4       → R2=7, R=7
根 = max(5, ≤4, 7) = 7（最善 R）。葉訪問 9/12、βカット1・αカット1
```

- βカット・αカット・**内部節点ごとの刈り**（M2 の 2 葉が丸ごと消える）が全部1本で見える
- 子順を逆（R→M→L）にすると α=7 が先に立ち、L2 も刈れて訪問数がさらに減る
  → 並べ替えで刈りが変わる比較が可能（子順反転トグルの追加を推奨、なくても
  木ファイル編集で再現可）

**→ 対応済み 2026-07-19**（提案どおり深さ3・葉12標準木に差し替え。葉訪問9/12・βカット1・αカット1を検証スクリプトで確認）

### 2.【中】Min-Max: 深さ2では再帰の核心が見えない

- 対象: `js/maps/minimax-tree.js`
- 深さ2の「min の表から max を取る」は静的な表読みで完結し、**MIN の下にまた MAX が
  いる**という交互再帰（「相手の応手の中に自分の最善がある」）が現れない。
  コールスタック可視化（実装済み）も最大2段しか積まれず宝の持ち腐れ。
- **提案**: 指摘1の深さ3標準木を Min-Max にも共通採用（枝刈りなしで全12葉を評価）。
  同じ木で「Min-Max 12葉 → α-β 9葉」の直接比較が成立し、正本 §2 の比較可能性
  原則がデモ間で通る。

**→ 対応済み 2026-07-19**（α-β と同型・同値の深さ3標準木を採用。根 v=7・全12葉評価を検証）

### 3.【中】モンテカルロ: 深さ2では「プレイアウト」が1回の乱択と区別できない

- 対象: `js/maps/monte-carlo-tree.js`
- 「終局までランダムに打つ」が、経路長2では実質サイコロ1回。HANDOFF に記録された
  設計「1ステップ=経路1マス」も経路が短すぎて意味を持たない。
- **提案**: 深さ3標準木を共通採用。プレイアウトが3手の系列として見え、
  手計算 EV も付記可能（提案木で 根EV≈4.92 vs Min-Max=7。MC 評価では
  L≈6.0 と R≈6.25 が僅差になり、Min-Max の 5 vs 7 と食い違う——「乱択平均は
  相手最善を仮定しない」を数値で示せる、現行より強い教材例）。

**→ 対応済み 2026-07-19**（深さ3標準木採用。真の乱択EV≈4.917を検証スクリプトで確認）

### 4.【軽〜中】AND-OR: 2段の OR-of-AND は真理値表と等価

- 対象: `js/maps/and-or-tree.js`
- 現行木（OR → AND → 葉）は DNF そのもので、交互再帰が見えない。
- **提案**: 葉「鍵を入手」を OR 節点（「買う」/「盗む」）に置き換えて深さ3へ。
  AND の子に選択肢が入り、「AND の中の OR」という再帰構造と部分打ち切りが見える。
  変更は木ファイルの1節点差し替えで済む。

**→ 対応済み 2026-07-19**（提案どおり「鍵を入手」を OR節点・買う→盗むに差し替え。根の結論は不変）

### 5.【軽】DFS: 初期地図が開けすぎてバックトラックがほぼ起きない

- 対象: `js/maps/dfs-map.js`
- 壁は上部3行のみで下半分は完全な空地。DFS の教材核はコールスタックの
  push/**pop**（行き止まり→戻る）だが、この地図では長蛇の蛇行が主になり
  pop がドラマにならない（実機で要確認だが地図構造から強く推定）。
- **提案**: 行き止まり2〜3本を持つ小さめの迷路に差し替え。地図サイズを一回り
  縮めてステップ数も短縮。

**→ 対応済み 2026-07-19**（行き止まり3本の小さめ迷路に差し替え。右→下→左→上のDFSでバックトラック3回・G到達をシミュレーションで検証）

### 6.【軽】最良優先: 「h だけだと悪い経路を掴む」古典例（凹型障害物）がない

- 対象: `js/maps/best-first-map.js`
- 現行はコスト2の直進帯 vs コスト1の迂回路で、弱点が**コスト面のみ**で示される
  （歩数的には最適経路を返してしまう）。greedy が凹型（コの字）障害物のポケットへ
  突っ込み、遠回り経路を確定する古典的な失敗例が無い。
- **提案**: S–G 間に G 側へ口を開けた凹型壁を置くプリセットを追加（差し替えでも可）。
  探索範囲の広がり（A* との対比）も視覚的に強くなる。

**→ 対応済み 2026-07-19**（S側に口を開けた凹型ポケットに差し替え。greedy がポケット10マスを全展開してから離脱し g=27 vs 真の最適22を返すことをシミュレーションで検証。歩数はBFS最短と同じ21だが、コストで明確な劣化を確認）

### 7.【軽】バンディット: 腕の設定が易しすぎて探索/活用ジレンマが弱い

- 対象: `js/maps/bandit-config.js`
- 最適腕 D（μ=0.8）と次点（0.55）のギャップが 0.25 と大きく、ε-greedy でも即収束。
  UCB1 の優位も、既定 100 手では乱数次第で逆転が見えにくい。
- **提案**: 「易しい（現行）」「難しい（例: 0.50/0.55/0.60 が接近）」の2プリセット化
  ＋既定手数を 300〜500 へ。可能なら同一シードで ε-greedy / UCB1 の並走比較
  （実装工数があるため、まずはプリセット追加が最小改修）。

**→ 対応済み 2026-07-19**（easy/hard 2プリセット化・既定手数300に変更。UI に難易度 select を追加。同一シードでの並走比較は未実装のまま次の課題）

### 8.【問題なし】

- **BFS / ダイクストラ**: 同一地図で「歩数最短（コスト20の直進） vs コスト最短
  （0コスト迂回）」が分かれる配置は設計として良い。据え置きで可。
- **A***: 3回の改訂を経て、負コスト帯（admissibility 崩し）まで持つ。良好。
- **AABB**: 二重実装比較・複雑度表つき。説明特化として十分。
- **FSM**: 6状態9イベント・遷移表・履歴。oneshot として十分（階層化・ガード条件は
  発展課題として解説パネルに1行追記でよい）。

### 9.【横断・既知の再確認】疑似コード現在行の同期が全デモ未実装

- 正本 §3 の共通 UI 必須項目「疑似コード表示と現在行の強調」「1ステップ**戻る**」は
  全デモ未実装（ROADMAP Phase B 残項目として既載）。木の深さ問題の次に
  アルゴリズム理解へ効く投資であることを、本レビューの観点からも再確認しておく。

**→ 未対応（バックログ）**。本改訂（2026-07-19）の対象外。platform 案件として別ブランチで対応する方針は変わらず。

## 推奨対応順（実装側への引き継ぎ）

1. **深さ3標準木の導入**（指摘1〜3を一括解決）: `minimax-tree.js` /
   `alpha-beta-tree.js` / `monte-carlo-tree.js` を同型の深さ3木に差し替え。
   コード変更は原則不要（データ差し替え＋コメントの手計算更新）。
   成熟度 oneshot → revised・修正+1 の対象（Phase C「教材改訂」の実体）
2. AND-OR の1節点差し替え（指摘4）
3. DFS 迷路化・最良優先の凹型障害物・バンディット難プリセット（指摘5〜7）
4. 疑似コード同期・戻る（指摘9）は platform 案件として別ブランチ

※ 1〜3 は木・地図データの差し替えのみで、`js/platform` にも各デモ JS にも
ほぼ手を入れずに済む。SPEC の無い実装先行トピック（経路5本）は CATALOG の
改訂メモ更新で足りる。
